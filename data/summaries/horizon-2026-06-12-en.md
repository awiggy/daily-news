## [AMD RCE Vulnerability Unfixed with CRC-32 Check](https://mrbruh.com/amd2/) ⭐️ 8.0/10

A remote code execution (RCE) vulnerability in AMD software remains unfixed, with the company's patch only adding a CRC-32 integrity check instead of proper cryptographic signature verification, making exploitation trivial if the webserver is compromised. This highlights AMD's inadequate security practices, potentially exposing millions of users to remote attacks. The failure to implement proper signature verification undermines trust in AMD's software security and could have serious consequences for enterprise and consumer systems. The vulnerability allows authenticated RCE via argument injection, and AMD's fix only uses a CRC-32 check, which is not cryptographically secure and can be easily bypassed. The researcher noted that while HTTPS prevents man-in-the-middle attacks, a compromised webserver still enables trivial exploitation.

hackernews · MrBruh · Jun 11, 16:03 · [Discussion](https://news.ycombinator.com/item?id=48492215)

**Background**: CRC-32 is an error-detecting code used to detect accidental data corruption, but it is not cryptographically secure and can be easily forged by an attacker. Proper signature verification uses cryptographic hashes (e.g., SHA-256) and digital signatures to ensure both integrity and authenticity. AMD's use of CRC-32 instead of a cryptographic signature means that an attacker who compromises the webserver can replace the executable with a malicious one while maintaining the same CRC-32 value.
